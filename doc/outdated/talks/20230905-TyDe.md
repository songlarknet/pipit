# Pipit: reactive systems in F*

## Reactive systems

[picture: car, train, plane, coffee machine]

Reactive systems are systems that must constantly react to changes in their environment, and they usually have strict deadlines on the latency.
In contrast to a program that computes a function and terminates -- a traditional compiler, for example -- reactive systems are usually expected to keep executing and responding to their environment indefinitely.

Such systems are everywhere, and many of them are safety-critical: cars, planes, trains, and motorcycles, are all reactive systems in some sense.

We can usually construct big reactive systems by composing together smaller reactive systems.
One such small reactive system is the driver that reliably extracts information from the accelerometer and gyroscope sensors in an autonomous vehicle.
Similar drivers also show up in less critical systems -- such as the wifi driver on my laptop.

## A sensor driver
These sorts of drivers are usually written in a low-level language such as C.
Let's have a look at an example of such a driver that tries to reads a temperature from a thermometer once per second, keeps track of when the last successful read was, and tries to reset the device after a certain number of failures.

```
sensor_driver() {
    int    age = MAX_AGE;
    sample bc  = SAMPLE_INVALID;

    while (true) {
        sample s = sensor_read();

        if (SAMPLE_VALID(s)) {
            bc  = s;
            age = 0;
        } else {
            age = age + 1;
        }

        if (age >= MAX_AGE) {
            bc = SAMPLE_INVALID;
        }
        broadcast(bc);

        if (age == RESET_AGE) {
            sensor_reset();
        }

        sleep(1000ms);
    }
}
```

The program has an infinite loop. Once a second, we use `sensor_read` to read a sample from the sensor then check if it's valid.
If it's valid, we use it as our most recent valid sample.
Otherwise, we increment the age of the current sample.
If the last valid sample is too old, we mark it as invalid and use `broadcast` to tell the driver's consumers what we know about the current sensor state.
After a certain number of bad reads, we reset the sensor.
Finally, we sleep until the next second.

### Partial specification

> If the driver broadcasts a valid sample, then the sample corresponds to a past sensor reading

Informally, this is a desirable property.

### An alternative: synchrony

```
sensor_driver⬆  (sensor_read: stream sample):
    { broadcast:    stream sample;
      sensor_reset: stream bool }
```

Specification
```
spec: \forall sensor_read: stream sample.
    let (bc, rst) = sensor_driver\lift sensor_read in
    \forall t. SAMPLE_VALID(bc t) ==>
        \exists t' <= t. bc t = sensor_read t'
```

A little bit stricter:
```
spec: \forall sensor_read: stream sample.

        \exists t' <= t. bc t = sensor_read t'
        /\  \not \exists t'' > t'. SAMPLE_VALID (sensor_read t'')
```

### The driver as a stream transformer
```
sensor_driver⬆  (sensor_read: stream sample) =
  let rec age t =
    if SAMPLE_VALID (sensor_read t) then 0
    else if t > 0 then age (t - 1) + 1
    else MAX_AGE
  in
  let rec bc t =
    if SAMPLE_VALID (sensor_read t) then sensor_read t
    else if age t < MAX_AGE then bc (t - 1)
    else SAMPLE_INVALID
  in
  let rst t = age t == RESET_AGE
  in { broadcast = bc; reset = rst }
```

### Synchronous impl
```
sensor_driver⬆  (sensor_read: stream sample) =
  let rec age t =
    if SAMPLE_VALID sensor_read then 0
    else (MAX_AGE `fby` age)
  in
  let rec bc =
    if SAMPLE_VALID sensor_read then sensor_read
    else if age < MAX_AGE then (SAMPLE_INVALID `fby` bc)
    else SAMPLE_INVALID
  in
  let rst = age == RESET_AGE
  in { broadcast = bc; reset = rst }
```

### The C code
```
sensor_driver() {
    sensor_driver_lift_state state;
    sensor_driver_lift_init(&state);

    while (true) {
        sample s = sensor_read();
        sensor_driver_result result = sensor_driver_lift_step(s, &state);

        broadcast(result.bc);

        if (result.rst) {
            sensor_reset();
        }

        sleep(1000ms);
    }
}
```

# Appendix

### A questionable specification
This driver is very simple, but if we want to verify it correct, it has some complexities.
The driver itself is sequential, but it runs concurrently with the sensor itself.
How do we specify `sensor_read` and `sleep`?

```
ghost now: nat;
ghost reading: nat -> sample;

sensor_read() { \result. result == reading[now] }
sleep(t) { now += t }
```
